# A-KEN Quest — UX / User-Journey Audit (2026-06-07)

Run because the CEO (msg 721) observed the dev loop was catching CONTENT bugs but
MISSING screen-structure/UX/journey defects. A world-class UX auditor walked all
key `?preview=` routes on the live demo (screenshots) + read the backing code.

**Headline: the audit immediately caught a CRITICAL defect the content-focused
loop had missed — four core screens render BLANK GREY (crash).** This validates
the CEO's concern and the value of a periodic UX walkthrough.

---

## P0 — broken / blocks core journey

### P0-1 — Four core screens crash to blank grey (CONFIRMED) → task #24
`battle`, `worldmap`, `achievements`, `parent` render an empty grey page and throw
a runtime type-cast error on first frame (`TypeError: type 'X' is not a subtype of
'Y'`, minified). Verified live: `?preview=battle` is blank grey (screenshot
logs/battle_check.png).
- **Impact is central, not edge:** Battle = the core FSRS vocab practice loop;
  Parent Dashboard = the screen parents pay for; Achievements = the reward loop.
- The 3 DqScene screens that DON'T crash (kotobahome, questmap, explore) avoid
  touching Firebase/Auth in init; the 4 broken ones each hit a Firebase-backed
  service (AuthService / XpService / AchievementService / progress) in init/build.
  battle wraps `getOrCreateUid()` in try/catch, so the throwing cast is a SEPARATE
  synchronous path (a `... as T` on an offline/null Firebase value) that escapes
  the guard and blanks the whole screen.
- **Fix (next iteration):** reproduce with Firebase offline; find + guard the
  unguarded cast(s); ensure a non-null fallback so the loading-guard path renders.
  THEN add the structural guard below.
- **Structural guard (closes the CEO's gap):** a smoke test that renders EVERY
  `?preview=` route with Firebase unavailable and FAILS on a blank/crash — so this
  whole defect class can never regress silently again.

### P0-2 —「ちずをみる / World Map」shows a level list, not a map (CONFIRMED) → task #22 (exists)
Confirmed + worse: three screens call themselves some "map" (KotobaHome button →
QuestMapScreen titled "World Map" but body is a vertical grade/town list;
WorldMapScreen = a zone list). None is the painted overworld — even though the
painted town art already exists in SceneView. Fix: build a real painted overworld
(towns as pins → SceneView) OR relabel so label==behavior; stop two screens both
claiming "World Map".

## P1 — confusing / missing-expected

- **P1-1 → #25 Brand leak:** onboarding header says "ENG Quest へようこそ" (old
  codename) — the first screen after Start breaks the ことばの勇者 / A-KEN Quest brand.
- **P1-2 → #26 Dev text shown to children:** the `speaking` screen renders
  "【開発中】… Azure Pronunciation Assessment接続後に本採点 … wires in via /v1/…" —
  internal TODO visible to a 6–15-yo. Gate behind kDebugMode or remove.
- **P1-3 → #28 Grade-label inconsistency:** 英検5級 / 英検3きゅう / 英けん2きゅう appear
  across screens — three spellings of the same thing. Pick one convention per
  audience, apply app-wide.
- **P1-4 → #27 No settings/mute/how-to-play in the journey:** the daily home +
  SceneView expose no mute (critical for an audio app + the coming BGM), no parent
  gate, no how-to-play. Add a persistent gear affordance.
- **P1-5 → #29 Hotspot discoverability:** NPC hotspots render as faint grey ghosts
  with no idle cue (only the coin pulses); a young child may not know they're
  tappable. Add a gentle idle pulse + a first-time coach-mark.
